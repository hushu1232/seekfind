# CLAUDE.md — 求问项目指南

## 项目上下文

求问是一个本地化的 AI 网页引导助手，使用 FastAPI + LangGraph + Chrome Extension 技术栈。

核心模块：
- `backend/agent_engine.py` — Agent 引擎（LangGraph StateGraph）
- `backend/tools/` — 12 个工具（检索/高亮/视觉/浏览器控制）
- `backend/memory/` — 三层记忆（短期/长期/持久化）
- `extension/` — Edge/Chrome 扩展（悬浮球 + 侧边栏 + Content Scripts）

## Karpathy 编码准则

> 来源：[Andrej Karpathy 的 LLM 编码观察](https://x.com/karpathy/status/2015883857489522876)

### 1. 先思考再编码

**不假设。不隐藏困惑。暴露权衡。**

- 明确说明假设。不确定时询问。
- 存在多种解读时，全部列出——不要默默选择。
- 有更简单的方案时，说出来。必要时反驳。
- 不清楚时停下。说出困惑之处，提问。

### 2. 简单优先

**解决问题的最少代码。不投机。**

- 不添加未要求的功能。
- 不为单次使用的代码创建抽象。
- 不为未要求的"灵活性"或"可配置性"添加代码。
- 不为不可能的场景添加错误处理。
- 200 行能写成 50 行就重写。

自问："高级工程师会觉得这过度复杂吗？"是就简化。

### 3. 精准修改

**只动必须动的。只清理自己制造的垃圾。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式。
- 不重构未损坏的部分。
- 匹配现有风格，即使你会用不同方式。
- 发现无关死代码时提及——但不删除。

你的改动产生孤儿代码时：
- 移除你的改动导致未使用的导入/变量/函数。
- 除非被要求，不移除预先存在的死代码。

测试：每一行改动都应直接追溯到用户请求。

### 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后通过"
- "修复 bug" → "写一个复现它的测试，然后通过"
- "重构 X" → "确保重构前后测试通过"

多步任务需说明计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
```

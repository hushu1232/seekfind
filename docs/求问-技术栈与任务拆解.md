# 第二部分：劣势攻克方案

## 劣势一：LLM 能力天花板（🔴 高）

### 问题本质

本地 7B 模型在复杂推理、多步工具调用、长上下文理解上不如 GPT-4o/Claude。

### 攻克方案：三层模型策略

```
┌─────────────────────────────────────────────────────────┐
│                    模型选择三层策略                        │
│                                                          │
│  Layer 1: 本地 7B（默认，零成本）                         │
│  ├─ qwen2.5:7b — 中文优秀，工具调用稳定                  │
│  ├─ 适用：日常文档问答、简单操作引导                      │
│  └─ 硬件要求：8GB 显存 / 16GB 内存                       │
│                                                          │
│  Layer 2: 本地大模型（可选升级）                          │
│  ├─ qwen2.5:14b — 推理质量显著提升                       │
│  ├─ qwen2.5:32b — 接近 GPT-4 水平                       │
│  ├─ 适用：复杂操作引导、多步推理                         │
│  └─ 硬件要求：14B 需 16GB 显存，32B 需 24GB+             │
│                                                          │
│  Layer 3: 云端 API（可选兜底）                           │
│  ├─ OpenAI GPT-4o / Claude / DeepSeek                   │
│  ├─ 适用：本地模型无法处理的复杂场景                      │
│  └─ 代价：需联网 + API Key + 付费                        │
└─────────────────────────────────────────────────────────┘
```

### 实现细节

**配置项（`.env`）：**

```env
# 模型策略：local | cloud | hybrid
MODEL_STRATEGY=hybrid

# 本地模型
OLLAMA_BASE_URL=http://localhost:11434/v1
OLLAMA_MODEL=qwen2.5:7b

# 云端模型（hybrid 模式下的兜底）
CLOUD_API_BASE_URL=https://api.openai.com/v1
CLOUD_API_KEY=
CLOUD_MODEL=gpt-4o-mini

# 自动降级阈值：本地模型连续 N 次工具调用失败后切换云端
FALLBACK_THRESHOLD=3
```

**LangGraph 中的降级逻辑：**

```
用户提问 → 本地 7B 推理
     │
     ├─ 工具调用成功 → 返回结果
     │
     └─ 连续失败 ≥ 3 次
          │
          ├─ MODEL_STRATEGY=hybrid → 自动切换云端模型重试
          │
          └─ MODEL_STRATEGY=local → 返回降级提示：
             "本地模型暂时无法处理此问题，建议切换更大模型或启用云端模式"
```

**用户界面切换：**

```
┌─────────────────────────┐
│  设置 > AI 模型          │
│                         │
│  当前模型：qwen2.5:7b   │
│                         │
│  [切换到 14B]  [云端模式] │
│                         │
│  ⚠️ 云端模式需联网       │
│     且会消耗 API 额度    │
└─────────────────────────┘
```

## 劣势二：首次配置门槛（🔴 高）

### 问题本质

用户需要：安装 Docker → 安装 Ollama → 拉取模型 → 启动容器 → 安装扩展 → 构建索引。6 步操作，普通用户无法完成。

### 攻克方案：三级安装体验

```
┌─────────────────────────────────────────────────────────┐
│                    安装体验三级策略                        │
│                                                          │
│  Level 1: 一键安装脚本（目标：双击即用）                   │
│  ├─ install.bat (Windows) / install.sh (Mac/Linux)       │
│  ├─ 自动检测环境 → 自动安装缺失依赖 → 自动拉取模型        │
│  ├─ 自动启动 Docker Compose                              │
│  └─ 最后输出："打开 Chrome，加载扩展目录即可使用"          │
│                                                          │
│  Level 2: 预打包 Docker 镜像（目标：docker pull 即用）     │
│  ├─ 镜像内预装 Python 依赖                               │
│  ├─ 启动时自动检测 Ollama → 自动拉取模型                  │
│  └─ 用户只需：docker compose up -d                       │
│                                                          │
│  Level 3: 桌面安装包（目标：双击 .exe 即用）               │
│  ├─ Electron/Tauri 打包的桌面应用                        │
│  ├─ 内嵌 Ollama 运行时                                   │
│  ├─ 内嵌 Docker Desktop 检测与引导                       │
│  └─ 图形化向导：下一步 → 下一步 → 完成                   │
└─────────────────────────────────────────────────────────┘
```

### install.bat 实现逻辑

```bat
@echo off
echo ========================================
echo   求问 - 本地智能网页引导球 安装程序
echo ========================================
echo.

:: Step 1: 检测 Docker
echo [1/6] 检测 Docker...
docker --version >nul 2>&1
if errorlevel 1 (
    echo ❌ 未检测到 Docker Desktop
    echo 请先安装 Docker Desktop: https://www.docker.com/products/docker-desktop/
    echo 安装后重新运行此脚本
    pause
    exit /b 1
)
echo ✅ Docker 已安装

:: Step 2: 检测 Ollama
echo [2/6] 检测 Ollama...
ollama --version >nul 2>&1
if errorlevel 1 (
    echo ⚠️ 未检测到 Ollama，正在自动安装...
    curl -L https://ollama.com/download/OllamaSetup.exe -o OllamaSetup.exe
    start /wait OllamaSetup.exe /S
    del OllamaSetup.exe
)
echo ✅ Ollama 已安装

:: Step 3: 启动 Ollama 服务
echo [3/6] 启动 Ollama 服务...
start /b ollama serve
timeout /t 3 >nul

:: Step 4: 拉取模型
echo [4/6] 拉取 AI 模型（约 4GB，首次需要下载）...
ollama pull qwen2.5:7b
ollama pull nomic-embed-text
echo ✅ 模型就绪

:: Step 5: 启动服务
echo [5/6] 启动求问服务...
docker compose up -d
echo ✅ 服务已启动

:: Step 6: 提示安装扩展
echo [6/6] 安装浏览器扩展...
echo.
echo ========================================
echo   安装完成！请按以下步骤操作：
echo ========================================
echo.
echo   1. 打开 Chrome 浏览器
echo   2. 地址栏输入: chrome://extensions
echo   3. 开启右上角"开发者模式"
echo   4. 点击"加载已解压的扩展程序"
echo   5. 选择本目录下的 extension 文件夹
echo.
echo   然后打开任意网页，点击侧边栏的求问球即可使用！
echo.
pause
```

## 劣势三：引导精确度（🟡 中）

### 问题本质

AI 推理的高亮位置可能不准，不如 WalkMe 预录制步骤精确。

### 攻克方案：五层精确度保障

```
精确度保障层级（从高到低）：

Layer 5: 预录制操作流（精确度 ★★★★★）
  │  用户手动录制的操作序列，回放时 100% 精确
  │  适用：固定流程（如"提交周报"）
  │
Layer 4: 元素指纹库（精确度 ★★★★☆）
  │  已知页面的元素 selector + 描述映射
  │  适用：经常访问的页面
  │
Layer 3: 选择器推理（精确度 ★★★☆☆）
  │  LLM 根据页面类型 + 文档描述推理出 selector
  │  适用：有标准结构的页面（如 Ant Design 表单）
  │
Layer 2: 视觉定位（精确度 ★★☆☆☆）
  │  moondream 截图分析获取坐标
  │  适用：选择器推理失败时
  │
Layer 1: 截图标注（精确度 ★☆☆☆☆）
  │  在截图上画红圈箭头，用户自行对照
  │  适用：所有方法都失败时的兜底
```

### 元素指纹库自动构建

```
用户首次成功操作某个按钮后，自动记录指纹：

  {
    "url_pattern": "https://app.example.com/dashboard*",
    "selector": "#create-project-btn",
    "xpath": "/html/body/div[1]/header/button[2]",
    "description": "创建新项目按钮",
    "visual_hash": "a3f2b1...",  // 元素截图的感知哈希
    "first_seen": "2026-05-30T10:00:00Z",
    "last_seen": "2026-05-30T10:00:00Z",
    "success_count": 3  // 成功定位次数
  }

下次用户问"在哪创建项目"：
  1. 查指纹库 → 命中 → 直接高亮（精确度 ★★★★☆）
  2. 未命中 → 走 AI 推理 → 视觉定位 → 截图标注
```

### 引导精确度反馈机制

```
每次引导后，球体气泡底部显示：

  ┌─────────────────────────┐
  │  第 1 步：点击「新建」    │
  │                         │
  │  [✅ 指对了]  [❌ 指错了] │
  └─────────────────────────┘

用户点击"指错了"：
  → 记录到反馈日志
  → 下次同一问题优先用其他定位方式
  → 累计错误 ≥ 3 次 → 标记该元素需要人工确认
```

## 劣势四：冷启动问题（🟡 中）

### 问题本质

文档索引为空时，search_docs 返回空结果，Agent 无法回答任何文档相关问题。

### 攻克方案：四层冷启动策略

```
┌─────────────────────────────────────────────────────────┐
│                    冷启动策略                              │
│                                                          │
│  Layer 1: 内置通用操作常识（零配置可用）                   │
│  ├─ 预置 50+ 常见产品的操作知识片段                       │
│  │  ├─ GitHub（创建仓库、提交 PR、设置 Actions）          │
│  │  ├─ GitLab / Jira / Confluence                       │
│  │  ├─ VS Code / IntelliJ                               │
│  │  ├─ Chrome / Firefox 设置                             │
│  │  ├─ Docker / Kubernetes 基础操作                      │
│  │  └─ 常见 SaaS 产品（飞书/钉钉/Notion/Figma）          │
│  ├─ 打包在 Docker 镜像内，首次启动自动加载                │
│  └─ 精确度一般，但"有回答"优于"无回答"                    │
│                                                          │
│  Layer 2: 文档导入向导（引导用户导入）                     │
│  ├─ 首次启动检测到索引为空 → 弹出导入向导                 │
│  ├─ 支持三种导入方式：                                    │
│  │  ├─ 输入文档站点 URL → 自动爬取                       │
│  │  ├─ 拖拽本地文件（Markdown/HTML/PDF）                 │
│  │  └─ 粘贴文档文本                                      │
│  └─ 导入后自动构建索引（进度条显示）                      │
│                                                          │
│  Layer 3: 浏览历史自动学习（被动积累）                     │
│  ├─ 用户浏览页面时，Content Script 自动提取页面正文       │
│  ├─ 经过隐私脱敏后，异步发送到后端建索引                  │
│  ├─ 用户无感知，逐步积累知识库                            │
│  └─ 需用户授权，默认关闭                                  │
│                                                          │
│  Layer 4: 联网搜索兜底（可选）                            │
│  ├─ 本地索引无结果时，提示用户：                          │
│  │  "本地文档未找到答案，是否联网搜索？"                  │
│  ├─ 用户确认后 → 调用 Bing/DuckDuckGo API               │
│  └─ 结果缓存到本地索引，下次不再联网                      │
└─────────────────────────────────────────────────────────┘
```

## 劣势五：浏览器兼容性（🟡 中）

### 问题本质

Chrome / Edge / Firefox 对 Manifest V3 的实现有差异，尤其在 Side Panel、Content Script 注入时机、API 可用性方面。

### 攻克方案：兼容层 + 逐浏览器适配

```
┌─────────────────────────────────────────────────────────┐
│                    浏览器兼容策略                          │
│                                                          │
│  1. Manifest V3 标准化                                    │
│  ├─ 使用 Manifest V3 最小子集（三家都支持的 API）         │
│  ├─ 避免使用 Chrome 独有 API（如 chrome.sidePanel）       │
│  └─ 用 polyfill 填补差异                                 │
│                                                          │
│  2. 差异点清单                                            │
│  ├─ Side Panel:                                          │
│  │  ├─ Chrome: chrome.sidePanel API                      │
│  │  ├─ Edge: 同 Chrome（Chromium 内核）                  │

│                                                          │
│  3. 适配层代码结构                                        │
```

### browser-compat.ts 接口设计

```typescript
// extension/common/browser-compat.ts

/**
 * 获取当前浏览器类型
 */
export function getBrowserType(): "chrome" | "edge" | "firefox" | "unknown"

/**
 * 获取侧边栏 API
 * Chrome/Edge: chrome.sidePanel
 * Firefox: chrome.sidebarAction (降级为 popup)
 */
export function getSidebarAPI(): SidebarAPI

/**
 * 截取当前标签页可见区域
 * 统一封装 chrome.tabs.captureVisibleTab

export async function captureVisibleTab(): Promise<string> // base64

/**
 * 在 document_start 时机注入脚本
 * Chrome/Edge: 直接执行

export function injectAtStart(fn: () => void): void
```

# 第四部分：验收标准

## 每阶段验收 Checklist

### Phase 1 验收

- [ ] `docker compose up -d` 后服务自动启动
- [ ] `curl /health` 返回 ok
- [ ] 扩展可加载，侧边栏显示对话面板
- [ ] 输入"怎么创建项目"→ 从文档索引检索 → 返回回答

- [ ] 密码字段输入不被采集
- [ ] 多标签页同时使用不冲突

### Phase 2 验收

- [ ] 问"在哪里新建"→ 页面上出现高亮框
- [ ] 选择器未命中 → 自动降级到视觉定位
- [ ] 视觉定位置信度低 → 自动降级到截图标注
- [ ] 截图标注在气泡中正确显示
- [ ] 用户反馈"指错了"→ 记录到日志

### Phase 3 验收

- [ ] 3D 球体 6 种状态切换流畅
- [ ] 页面加载 <50ms 出现骨架球
- [ ] 关闭浏览器再打开 → 状态恢复
- [ ] 检测到表单页面 → 主动提示用户
- [ ] 录制操作流 → 回放为分步指引
- [ ] 元素指纹库命中时高亮速度 <100ms

### Phase 4 验收

- [ ] 说"小求小求"→ 球体进入监听状态
- [ ] 语音提问 → 识别为文本 → 检索 → 回答
- [ ] 回答朗读时球体嘴巴同步
- [ ] 说"小求，闭嘴"→ 立即停止
- [ ] 双击 install.bat → 10 分钟内完成安装
- [ ] 设置面板可切换本地/云端模型
- [ ] 隐私面板可查看/清除所有数据
